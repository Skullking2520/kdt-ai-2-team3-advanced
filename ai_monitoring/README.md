# LLM + RAG 응답 monitoring을 하기 위한 폴더

본 폴더는 llmops의 observability관련 파일들을 모아놓았습니다.

현재는 ai_service 폴더에 두기 위한 임시 폴더입니다.

# 🚀 Langfuse 개요 및 핵심 기능 정리

Langfuse는 LLM 애플리케이션의 개발, 모니터링, 그리고 테스트 단계를 가속화하기 위해 설계된 **오픈소스 기반의 LLM 엔지니어링 플랫폼(LLMOps)**입니다. 프로토타입 단계부터 대규모 엔터프라이즈 환경까지 LLM 앱의 전체 수명 주기를 효율적으로 관리할 수 있도록 돕습니다.

---

## 📌 핵심 기능 (Core Capabilities)

### 1. LLM 옵서버빌리티 & 애플리케이션 추적 (Observability & Tracing)

- **계층적 트레이싱:** 복잡하게 중첩된 LLM 호출, 도구 사용(Tool calls), RAG 검색 단계를 트리 구조(`Trace` - `Span` - `Generation`)로 시각화합니다.
- **성능 및 비용 모니터링:** 실행 속도(Latency), 토큰 소모량, 정확한 비용 지표를 실시간 대시보드로 추적하여 병목 구간을 진단합니다.
- **대화 세션 관리:** 멀티턴 대화나 에이전트 루프 구조의 연속적인 트레이스를 하나의 `Session`으로 묶어 사용자 경험 단위로 추적할 수 있습니다.

### 2. 프롬프트 관리 시스템 (Prompt Management)

- **코드 분리 및 중앙 관리:** 코드를 재배포할 필요 없이 Langfuse UI에서 프롬프트를 작성, 수정, 버전 관리할 수 있습니다.
- **에지 캐싱(Edge-Cached):** 프롬프트 가져오기 API 성능이 독립적으로 최적화되어 프로덕션 환경의 지연 시간을 최소화합니다.

### 3. 유연한 평가 체계 (Evaluation)

- **LLM-as-a-Judge:** 다른 고성능 모델(GPT-4o 등)을 자동 평가자로 설정하여 환각률, 컨텍스트 관련성 등을 정량적으로 측정합니다.
- **Ragas 연동 및 외부 지표 바인딩:** Ragas와 같은 전문 RAG 평가 프레임워크 스크립트 결과 지표를 특정 트레이스의 `Score` 객체에 간편하게 동기화합니다.
- **휴먼 인 더 루프(Human-in-the-Loop):** UI를 통해 평가 점수를 수동으로 입력하거나, 정답 데이터셋(Golden Dataset) 구축용 아노테이션 작업을 협업할 수 있습니다.

### 4. 오픈소스 및 강력한 데이터 제어 (Self-Hosting)

- **인프라 통제권:** 외부 SaaS 환경에 데이터를 전송하지 않고 사내 독립된 인프라(Docker, Kubernetes 등)에 완전히 격리된 배포가 가능하여 민감한 프롬프트 및 개인정보(PII) 누출을 원천 차단합니다.
- **비동기 설계:** 백그라운드 배칭 및 비동기 수집(Async Ingestion) 방식을 채택하여 모니터링 도구의 도입이 주 애플리케이션의 런타임 성능에 영향을 주지 않습니다.

## 🛠️ 통합 및 확장성 (Integrations)

- **프레임워크:** LangChain(Callback 연동), LlamaIndex, OpenAI SDK, LiteLLM 등 대부분의 핵심 AI 생태계 및 모델 지원
- **표준 프로토콜:** OpenTelemetry(OTel) 네이티브 기반 아키텍처 지원으로 기존 인프라 옵서버빌리티 도구(Datadog 등)와의 결합이 용이

## langfuse가 v4이상이 되면서 opentelemetry를 결합한 이유 (skip해도 됨)

Langfuse가 최신 버전(v3 베타를 거쳐 v4로 안착)으로 진화하며 핵심 아키텍처를 OpenTelemetry(OTel) 기반으로 전면 전환한 이유는 "성능 극대화"와 "생태계 확장(기능적 범용성)" 두 가지를 모두 잡기 위해서입니다.

1. 🚀 성능상의 이점 (Performance)

기존 v2 방식은 Langfuse가 자체 개발한 커스텀 HTTP 통신(전송) 라이브러리를 사용해 데이터를 보냈지만, v4부터는 전 세계 수많은 대기업이 검증한 OpenTelemetry의 고성능 엔진을 그대로 탑재했습니다.

배치 전송 효율화 (레이턴시 최소화): OTel의 고도화된 배치 엑스포터(Batched Exporter)를 활용하여 메인 애플리케이션의 성능(LLM 응답 속도 등)에 전혀 영향을 주지 않고 백그라운드에서 데이터를 매우 효율적으로 묶어 보냅니다.

데이터 모델 최적화 (S3 비용 및 탐색 속도 개선): v4에서는 데이터 모델이 '관측 중심(Observation-centric)' 불변 데이터 구조로 변경되었습니다. 데이터가 불변(Immutable) 상태로 저장되므로 데이터베이스(ClickHouse 등)에서 비싼 Join 연산을 할 필요가 없어졌고, 자체 구축(Self-hosted) 기준 S3 비용이 최대 85%까지 절감되는 극적인 성능 향상을 이뤄냈습니다.

UI 및 에이전트 대규모 쿼리 가속: AI 에이전트가 고도화되면서 하나의 트레이스 안에 수천 개의 복잡한 체인(LLM 호출, 툴 실행 등)이 엮이게 되는데, OTel 컨텍스트 덕분에 대규모 데이터도 UI 대시보드에서 지연 없이 실시간으로 빠르게 조회하고 필터링할 수 있게 되었습니다.

2. 🛠️ 기능 및 연동상의 이점 (Features)

그동안은 새로운 LLM 프레임워크나 라이브러리가 나올 때마다 Langfuse 개발팀이 전용 SDK나 래퍼(Wrapper)를 일일이 개발해야 했습니다. OTel 도입 이후에는 이 한계가 완전히 깨졌습니다.

타사 라이브러리 자동 캡처 (Auto-nesting): 내 코드에 Langfuse 코드만 심어두면, 함께 사용 중인 외부 라이브러리(예: Anthropic SDK, LlamaIndex 등)가 내뿜는 OpenTelemetry 스팬(Span)을 Langfuse가 자동으로 감지합니다. 별도 연동 코드를 짜지 않아도 트레이스 구조 내부에 예쁘게 정렬되어 들어옵니다.

## span 작성하는 3가지 방법: @observe, with문, langfuse.start_observation(name)

1. @observe() 데코레이터의 자동 종료함수 위에 @observe()를 붙이면, 파이썬이 해당 함수를 실행하기 직전에 스팬(Span)을 생성하고, 함수가 return 되거나 실행이 끝나는 순간 내부적으로 종료 매커니즘을 트리거하여 스팬을 닫습니다.

2. with 문(컨텍스트 매니저)의 자동 종료파이썬의 with 문은 **enter**와 **exit**라는 특수 메서드를 기반으로 동작합니다.

블록 진입 시: **enter**가 실행되면서 스팬을 시작합니다.

블록 탈출 시: **exit**가 실행되면서 수동 관리의 .end()와 동일한 역할을 하는 종료 코드를 자동으로 수행합니다.

#### ⚠️ 예외(Error) 발생 시에도 안전한가요?

네, 훨씬 더 안전합니다. 수동으로 .end()를 작성하면 코드 중간에 에러(Exception)가 터졌을 때 .end() 라인까지 도달하지 못해 스팬이 영원히 안 닫히는 버그가 생길 수 있습니다. (이를 막으려면 try...finally로 감싸야 합니다.)

반면 with 문과 @observe는 내부에서 에러가 발생하더라도 파이썬 인터프리터가 무조건 종료 처리를 보장합니다. 또한, 발생한 에러 메시지와 트레이스백(Traceback)을 Langfuse 대시보드에 자동으로 '실패(Error)' 상태로 기록해 주기 때문에 디버깅에도 훨씬 유리합니다.

## Langfuse 화면 공유 (접속) 여부

팀원이 동일한 랭퓨즈 프로젝트에 접속하여 대시보드와 데이터를 열람하려면 다음 두가지 방법을 사용할 수 있습니다.

1. 프로젝트 멤버 초대: 랭퓨즈 ui의 project setting -> members 메뉴로 이동하여 팀원의 이메일을 초대할 수 있습니다. 팀원은 본인의 랭퓨즈 계정으로 로그인하여 화면을 함께 확인하고 협업할 수 있습니다.

2. 트레이스 공유 링크(특정 화면 공유): 대화 기록(session)이나 특정 추적(Trace)화면만 공유하고 싶다면 우측 상단의 share버튼을 눌러 공용 링크(public link)를 생성해 팀원에게 전달할 수 있습니다.
