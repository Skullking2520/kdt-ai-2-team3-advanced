# 백엔드 위키 — 스미싱 탐지 서비스

> 팀원 누구나 읽고 발표에서 설명할 수 있도록 작성한 문서입니다.
> 최종 수정 2026-06-10

---

## 백엔드가 하는 일 (한 줄 요약)

> 사용자가 입력한 문자 / URL / 이미지 / 전화번호를 받아서 스미싱 여부를 판단하고, 결과를 프론트엔드에 돌려주고 DB에 기록한다.

---

## 전체 서비스 구조

```
[프론트엔드]  ──→  [백엔드 FastAPI]  ──→  [MySQL DB]
                        │
                        ├──→  [ai_service (모델팀)]
                        │        LangGraph + RAG
                        │
                        └──→  [CLOVA OCR]  (이미지 분석 시)
```

- **백엔드**: 요청 수신, 파이프라인 조율, DB 저장 담당
- **ai_service**: 모델팀이 별도 서버로 운영. 스미싱 판단과 이유 생성 담당
- **MySQL**: 분석 결과 전체를 로그로 저장
- **프론트엔드**와는 REST API로 통신

## 분석 파이프라인 (핵심)

사용자 입력이 들어오면 입력 유형에 따라 다른 경로를 탄다.

### 입력 유형별 경로

| 입력 유형 | 경로 |
|-----------|------|
| `sms` | 문자 메시지 → 정적 패턴 검사 → 인코더 → (스미싱이면) 디코더 |
| `url` | URL → 블랙리스트 검사만 (인코더 없음) |
| `image` | 이미지 → OCR 텍스트 추출 → SMS 파이프라인 |
| `phone` | 전화번호 → 블랙리스트 검사만 |

---

### SMS 파이프라인 (가장 핵심)

```
문자 입력
    │
    ▼
[1단계] 정적 패턴 검사
    DB에 등록된 악성 URL / 전화번호와 일치하는지 확인
    │
    ├── 일치함 → STATIC_PATTERN 판정 → 즉시 스미싱 반환
    │
    └── 불일치
            │
            ▼
        [2단계] 인코더 모델 호출
            문자 내용을 모델에게 보내서 스미싱 여부와 신뢰도 점수(0~1) 받음
            │
            ├── "정상" → ENCODER 판정 → 안전 반환
            │
            └── "스미싱"
                    │
                    ├── 신뢰도 ≥ 0.90 → ENCODER 판정
                    │                    (인코더 확신이 높음)
                    │
                    └── 신뢰도 < 0.90 → RAG_DECODER 판정
                                         (인코더가 애매함 → 디코더도 참여)
                    두 경우 모두 디코더를 호출해서 "왜 스미싱인지" 설명 생성
```

**판정 결과 3가지**

| 판정 | 의미 | 예시 상황 |
|------|------|-----------|
| `STATIC_PATTERN` | DB 블랙리스트 직접 매칭 | 알려진 악성 URL 포함 |
| `ENCODER` | 인코더 모델 단독 판정 | 모델이 스미싱 확신 (90% 이상) 또는 정상 |
| `RAG_DECODER` | 인코더 + 디코더 함께 판정 | 모델 신뢰도 애매 (90% 미만) |

---

### URL 파이프라인

URL은 인코더에 넣기 부적합하다. (URL 문자열 자체를 자연어 모델로 분류하면 정확도가 낮음)
그래서 **블랙리스트 매칭만** 수행한다.

```
URL 입력 → DB 블랙리스트 검사
    ├── 등록된 악성 URL → 스미싱 반환
    └── 없음 → 안전 반환
```

---

### 이미지 파이프라인

이미지는 먼저 텍스트로 변환(OCR)한 뒤 SMS 파이프라인을 그대로 탄다.

```
이미지(base64) → OCR 텍스트 추출 → SMS 파이프라인
```

OCR은 **PaddleOCR**을 먼저 시도하고, 품질이 낮으면 **CLOVA OCR**로 재시도한다.

---

### 전화번호 조회

`GET /api/sender/010-1234-5678` 형태로 요청한다.
DB에 신고된 번호인지만 검사한다. 문자 분석과 달리 별도의 엔드포인트로 제공된다.

## API 엔드포인트

| 메서드 | 경로 | 역할 |
|--------|------|------|
| `GET` | `/health` | 서버 상태 확인 |


## DB 구조 (핵심 3개 테이블)

### `smishing_logs` — 모든 분석 결과 기록

| 컬럼 | 설명 |
|------|------|
| `content` | PII 마스킹된 문자 내용 (원본 아님) |
| `is_smishing` | 최종 스미싱 여부 (true/false) |
| `detection_type` | 어떤 방식으로 판정했는지 (STATIC_PATTERN / ENCODER / RAG_DECODER) |
| `input_type` | 입력 유형 (SMS / URL / IMAGE / PHONE) |
| `ai_score` | 인코더 신뢰도 점수 (0.0~1.0) |
| `reasoning` | 디코더가 생성한 판정 이유 |

> PII 마스킹: 전화번호 → `<PHONE>`, URL → `<URL>`, 주민번호 → `<SSN>` 등으로 치환해서 저장

### `static_patterns` — 악성 URL / 전화번호 블랙리스트

관리자가 등록하거나 모델이 탐지한 악성 패턴을 저장.
1단계 정적 패턴 검사에서 이 테이블을 조회한다.

### `model_info` — AI 모델 버전 정보

어떤 버전의 인코더/디코더가 판정했는지 기록.

---

## 개인정보 보호 처리

사용자가 입력한 문자를 그대로 저장하지 않는다.

```
원본: "엄마 나야. 010-9876-5432로 연락줘"
저장: "엄마 나야. <PHONE>로 연락줘"
```

마스킹 항목: 전화번호, URL, 이메일, 주민번호, 계좌번호, 카드번호

## 프론트엔드와 연동 방식

프론트엔드는 `api.ts` 파일 하나에서 모든 API 호출을 관리한다.

```
[사용자 문자 입력] → Analyzer.tsx
    → api.analyze({ type: 'sms', content: '...' })
    → POST /api/predict
    → AnalysisResult.tsx (결과 화면)
```

이미지 분석은 결과를 화면 간 전달할 때 URL 파라미터가 아닌 **React Router state**를 사용한다.
→ 같은 분석 요청이 두 번 DB에 저장되는 것을 방지하기 위한 구조.

## OCR 품질 판단 기준 및 테스트 결과

### 두 단계 OCR 파이프라인

PaddleOCR(무료, 빠름)을 먼저 시도하고, 품질이 낮을 때만 CLOVA OCR(유료, 정확)로 재처리한다.

```
이미지
  │
  ▼
PaddleOCR 실행
  │
  ├── 품질 기준 통과 → PaddleOCR 결과 사용
  │
  └── 품질 기준 미달 → CLOVA OCR 재처리
                            │
                            ├── 성공 → CLOVA 결과 사용
                            └── 실패 → PaddleOCR 결과 유지 (최선)
```

### CLOVA fallback 판단 기준

아래 조건 중 하나라도 해당되면 CLOVA로 재처리한다.

| 기준 | 임계값 | 이유 |
|------|--------|------|
| 평균 신뢰도 (avg_confidence) | < 0.95 | 여러 문자가 불확실하게 읽힌 경우 |
| 최소 신뢰도 (min_confidence) | < 0.85 | 핵심 단어 하나가 통째로 잘못 읽힌 경우 |
| 종합 품질 점수 | < 0.90 | 한국어 비율·오타·특수문자 등 종합 감점 |
| OCR 노이즈 | 해당 시 | 한글+기호 혼재, 낱자 출현 등 명백한 오인식 흔적 |
| broken 패턴 | 해당 시 | URL·전화번호가 끊겨 스미싱 탐지 불가 우려 |

### 종합 품질 점수 감점 항목

| 항목 | 감점 | 근거 |
|------|------|------|
| 텍스트 길이 < 10자 | -0.5 | 텍스트 추출 자체 실패 |
| 한글 비율 < 0.45 | -0.5 | 한글이 절반 미만 → 전면 오인식 |
| 한글 비율 < 0.65 | -0.25 | 한글 비율 낮음 → 부분 오인식 |
| 특수문자({}\[\]\|^~■) 출현 | -0.2 | OCR이 한글을 기호로 잘못 읽은 흔적 |
| 특수문자 비율 > 5% | -0.2 | 광범위한 오인식 |
| 반복 텍스트 | -0.4 | 같은 영역 이중 인식 |
| 오타 의심 토큰 | -0.2~0.6 | 한글+영문 혼재, 낱자 출현 |
| broken URL/전화번호 | -0.3 | "h t t p", "0 1 0" 형태로 끊김 |

### 실제 테스트 결과 (2026-06-10, 로컬 M-시리즈)

**케이스 1 — 선명한 이미지 → PaddleOCR 단독 처리**

```
avg_conf=1.000  min_conf=0.999  quality=1.00
→ PaddleOCR 품질 양호 → PaddleOCR 결과 사용
추출: "[국외발신] 고객님 계좌가 정지되었습니다. 확인: http://kb-check.xyz"
```

**케이스 2 — 흐린 이미지 → CLOVA fallback**

```
avg_conf=0.960  min_conf=0.807  quality=0.20
→ 품질 기준 미달 → CLOVA fallback 시도

PaddleOCR: "급할 기능성이 높 ㄷ[. 3억원을 개인적으로 시[용하지 않았디면 모든 혐"
CLOVA:     "급할 가능성이 높다. 3억원을 개인적으로 사용하지 않았다면 모든 혐"
```

`기능성→가능성`, `시[용→사용`, `않았디면→않았다면` 등 CLOVA가 오인식 문자를 정확히 복원.

## 데이터팀 연동 현황

데이터팀은 S3 기반 배치 파이프라인을 구현 중이다. 백엔드(실시간 API)와 구조가 다르지만 같은 MySQL DB를 공유하기 때문에 스키마 합의가 필요하다.

### 실시간 vs 배치 구조 차이

| 구분 | 백엔드 | 데이터팀 |
|------|--------|----------|
| 처리 방식 | 사용자 요청 즉시 처리 | S3에 쌓고 단계별 배치 처리 |
| 로그 저장 | MySQL `smishing_logs` | MySQL `processing_log` + S3 파일 |
| S3 연동 | 없음 | raw → labeled → processed → reason 단계별 저장 |

### 협의 필요한 항목

**1. 테이블 명칭 통일**

| 테이블 역할 | 백엔드 | 데이터팀 |
|------------|--------|----------|
| 블랙리스트 | `static_patterns` | `blacklist` |
| 분석 로그 | `smishing_logs` | `processing_log` |

같은 MySQL을 공유하므로 어느 쪽 이름으로 통일할지 결정 필요.

**2. 스코어 단위 불일치**

| 항목 | 백엔드 | 데이터팀 |
|------|--------|----------|
| 스코어 범위 | `ai_score` 0.0~1.0 (float) | `score` 0~100 (int) |
| RAG 진입 임계값 | `< 0.90` | `40~70` |

단위를 맞추거나 저장 전 변환 로직을 명시해야 통계 집계 시 불일치가 없다.

**3. 블랙리스트 컬럼 차이**

데이터팀 `blacklist`에는 백엔드에 없는 컬럼이 있다.

| 컬럼 | 데이터팀 | 백엔드 |
|------|:--------:|:------:|
| `category` (택배사칭/금융사칭 등) | ✅ | ❌ |
| `source` (kisa/openphish 등) | ✅ | ❌ |
| `severity` (low/medium/high) | ✅ | ❌ |
| `report_count` | ✅ | ❌ |

데이터팀이 이 컬럼들을 채우는 구조라면 백엔드도 같은 테이블을 공유하거나 컬럼을 추가해야 한다.

## 모델팀 연동 현황

모델팀은 `ai_service`라는 별도 FastAPI 서버를 구현 중이다.

### 현재 합의된 연동 방식

**백엔드 → 모델팀 호출 (디코더)**

```
POST /api/v1/graph/invoke
{
  "text": "마스킹된 문자 내용"
}
```

**모델팀 응답 (현재)**

```json
{
  "is_smishing": true,
  "reason": "공공기관을 사칭해 URL 접속을 유도하므로 스미싱 가능성이 높습니다."
}
```

### 협의 중인 사항

| 항목 | 현재 상태 | 협의 필요 내용 |
|------|-----------|----------------|
| 응답 형식 | 모델팀 `reason` 필드 / 백엔드 기대값은 `type` + `explanation` | 유형 분류 필드 추가 여부 |
| 인코더 분리 여부 | 모델팀 LangGraph에 통합되어 있음 | 인코더 별도 제공 여부 |
| 서비스 포트 | 모델팀 8000 / 백엔드 8000 충돌 | 포트 조율 필요 |
